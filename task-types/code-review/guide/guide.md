# 代码审查任务创建指南

这个 guide 用于帮助创建 `code-review` 类型任务的 `task.md`。它是补充材料，不是强制模板。

## 任务边界

代码审查任务适合：

- 只读审查一个仓库、分支、PR、commit 范围或变更集
- 识别高风险问题，例如安全漏洞、崩溃级逻辑错误、严重性能问题
- 给出具体文件、行号、原因和修复建议

代码审查任务默认不适合：

- 直接修改代码
- 自动修复问题
- 只做代码风格、命名、格式偏好类检查

如果用户确实希望修改代码，应考虑使用 `feature-dev` 类型，或在任务中明确说明“允许修改”。

## 创建时建议澄清的信息

- 目标仓库或代码位置
- 审查范围：commit、分支、PR、时间范围、目录或文件集合
- 是否需要同步远程代码
- 是否只报告高风险问题，还是也报告中低风险和建议
- 是否需要按提交者、模块、风险等级或文件分组
- 是否需要检查测试覆盖、兼容性、性能、可观测性、安全合规
- 对输出格式的偏好

## task.md 推荐内容

可以自由组织，但建议包含：

- 审查对象：仓库、分支、commit 范围或路径
- 审查目标：希望发现哪类风险
- 权限边界：是否只读，是否允许拉取代码，是否允许运行测试
- 报告粒度：按 commit、按作者、按文件、按风险等级等
- 严重性标准：哪些问题必须报告，哪些可以忽略
- 成功标准：问题需要包含哪些证据和建议

## 风险等级建议

可按任务需要调整：

| 级别 | 示例 | 默认处理 |
| --- | --- | --- |
| 高风险 | 安全漏洞、敏感信息泄露、崩溃级逻辑错误、严重性能退化、数据损坏风险 | 必须报告 |
| 中风险 | 边界条件、错误处理缺失、潜在资源泄漏、测试缺口 | 视任务要求报告 |
| 低风险 | 命名、格式、注释、轻微风格偏好 | 默认不报告 |

## 执行建议

执行 agent 通常应：

- 保持只读，除非任务明确授权修改
- 先理解项目结构和相关上下文，再判断 diff
- 不只看 diff，必要时读取相关完整文件
- 只报告有明确证据的问题
- 每个问题尽量包含文件、行号、风险说明、触发条件和修复建议
- 不把“个人偏好”当成缺陷

## 报告结构建议

可参考以下结构，不强制：

```markdown
# 代码审查报告：{范围}

## 概述

## 审查范围

## 高风险问题

## 中风险问题

## 测试与验证观察

## 总结
```

## 不要继承的旧约定

- 不要求任务输入是固定 JSON 字段
- 不要求固定按作者分组，除非任务明确需要
- 不要求固定保存到 `reports/` 目录
- 不要求在 `task.md` 中重复系统结果协议
