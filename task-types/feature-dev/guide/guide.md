# 功能开发任务创建指南

这个 guide 用于帮助创建 `feature-dev` 类型任务的 `task.md`。它是补充材料，不是强制模板。

## 任务边界

功能开发任务适合：

- 在代码库中实现新功能、修复缺陷或重构局部模块
- 自主阅读代码、制定方案、修改文件并验证结果
- 在任务完成时说明变更内容、验证方式和风险

功能开发任务不适合：

- 只读审查，不改代码的场景；这种更适合 `code-review` 或 `doc-review`
- 需求仍然完全不明确、无法判断完成标准的场景
- 需要人工长期协作但没有明确阶段性目标的场景

## 创建时建议澄清的信息

- 目标代码库或工作目录
- 需要实现、修复或调整的具体行为
- 用户可观察的期望结果
- 验收标准和非目标
- 相关入口、页面、命令、API、测试或文档
- 是否有兼容性、性能、安全、风格或设计系统约束
- 是否允许新增依赖、迁移数据、修改配置或调整公开接口
- 期望执行哪些验证命令

## task.md 推荐内容

可以自由组织，但建议包含：

- 背景：为什么要做这个改动
- 目标：需要达成的用户价值或系统行为
- 范围：要改哪里，不改哪里
- 验收标准：如何判断完成
- 约束：技术、产品、兼容、安全、性能、时间等限制
- 已知上下文：相关文件、接口、错误日志、截图、设计稿等
- 验证建议：可运行的测试、构建、lint 或手工验证路径

## 执行建议

执行 agent 通常应：

- 先阅读任务和代码上下文，不急着改
- 对不明确但可合理推断的点做最小假设，并在结果中说明
- 优先做局部、可验证、可回滚的改动
- 不引入与任务无关的大型重构
- 不覆盖用户已有未提交改动
- 修改后运行与任务相关的最小必要验证
- 如果无法继续，使用 `paused` 或 `blocked` 说明原因，而不是硬做

## 输出建议

完成时建议说明：

- 实现了什么
- 修改了哪些主要区域
- 运行了哪些验证
- 还有哪些风险或未覆盖点
- 如果生成了关键产物，可在 `artifactRefs` 中引用相对当前工作目录的路径

## 不要继承的旧约定

- 不再有固定的 plan / implement / verify Phase
- 不需要按旧的人工审批阶段推进
- Orchestrator 不负责自动建 git 分支、自动跑验证命令或自动回滚
- 不要求固定保存方案到 `reports/` 目录
- 不要求在 `task.md` 中重复系统结果协议
