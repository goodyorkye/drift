# 通用任务创建指南

这个 guide 用于帮助创建 `general` 类型任务的 `task.md`。它适合那些暂时不想先细分到开发、调研、审查等专门类型的任务。

## 适用场景

通用任务适合：

- 还在探索中、暂时不方便严格归类的请求
- 需要 agent 根据材料先理解上下文，再决定如何展开的工作
- 一次性的整理、分析、说明、实现、校对或组合型任务

如果任务目标已经非常明确地属于某个专门类型，仍优先选择对应类型。

## 创建时建议澄清的信息

- 这次任务最终要完成什么
- 当前提供了哪些材料，还缺哪些材料
- 有哪些明确约束，例如时间、范围、风格、工具、输出格式
- 期望产出是什么，例如文档、代码修改、报告、清单或建议
- 哪些事情不要做
- 最终结果是否需要保存成可下载文件；如果没有明确说不要文件，默认按“需要产物文件”理解

## task.md 推荐内容

可以自由组织，但建议至少讲清楚：

- 背景：为什么要做这件事
- 目标：最终交付物或结论是什么
- 输入材料：当前目录里哪些文件最重要
- 约束：必须遵守的范围、限制和风格要求
- 成功标准：什么样算完成

## 执行建议

执行 agent 通常应：

- 优先阅读 `task.md` 和当前目录材料
- 先厘清任务目标，再决定具体做法
- 在信息不足时明确指出缺口或假设
- 不把运行时状态写回任务材料目录
- 如果任务目标本身是在生成内容结果，默认把最终结果写成当前工作目录中的文件，并在 `agent-result.json.artifactRefs` 中声明它，除非任务明确说不要文件产物

## 不要继承的旧约定

- 不要求固定 JSON 输入
- 不要求固定输出目录
- 不要求把系统状态、队列逻辑或结果协议写进 `task.md`
