# 文档审查任务创建指南

这个 guide 用于帮助创建 `doc-review` 类型任务的 `task.md`。它是补充材料，不是强制模板。

## 任务边界

文档审查任务适合：

- 只读检查文档目录或文档集合
- 发现事实错误、逻辑矛盾、过期信息、结构问题和格式问题
- 跨文档发现重复、冲突、缺口和可合并内容
- 产出审查报告或改进建议

文档审查任务默认不适合：

- 直接重写文档
- 自动修改文件
- 审查文档之外的代码实现细节，除非任务明确要求对照代码核验

## 创建时建议澄清的信息

- 要审查的目录、文件或文档集合
- 是否需要只看 Markdown，还是包括 PDF、Word、网页、表格等
- 审查重点：正确性、一致性、完整性、结构、格式、链接、术语统一
- 是否需要对照代码、产品行为、外部事实或设计文档核验
- 是否只输出问题，还是也输出重构建议和优先级
- 是否允许修改文档；默认应按只读处理
- 输出格式和深度偏好

## task.md 推荐内容

可以自由组织，但建议包含：

- 审查对象：具体目录、文件或文档范围
- 审查目标：希望发现什么问题
- 判断依据：以哪些文档、代码、产品事实或外部来源为准
- 权限边界：是否只读，是否允许修正文档
- 输出要求：是否需要表格、优先级、文件行号、修复建议
- 成功标准：什么样的审查结果算完成

## 审查维度建议

- 正确性：事实错误、技术错误、过期版本、错误示例
- 一致性：术语不统一、章节互相矛盾、同一概念多套说法
- 完整性：缺少前置条件、边界条件、失败场景、操作步骤
- 结构：层级混乱、重复内容、信息分散、导航困难
- 可读性：表达含糊、长段落过多、读者视角不清
- 格式：Markdown 语法、链接有效性、标题层级、表格可读性

## 报告结构建议

可参考以下结构，不强制：

```markdown
# 文档审查报告：{范围}

## 概述

## 审查范围

## 主要问题

## 文档间矛盾

## 结构与可读性建议

## 优先处理清单

## 质量良好的部分
```

## 不要继承的旧约定

- 不要求至少发现固定数量的问题；如果质量良好，应说明审查依据
- 不要求固定保存到 `reports/` 目录
- 不要求在 `task.md` 中重复系统结果协议
