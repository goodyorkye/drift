# Git 审计任务创建指南

这个 guide 用于帮助创建 `git-audit` 类型任务的 `task.md`。它是补充材料，不是强制模板。

## 任务边界

Git 审计任务适合：

- 审计仓库中的 commit、commit 范围、分支或周期性新增提交
- 逐 commit 检查安全、逻辑、架构、性能、可观测性和提交质量问题
- 产出可追溯到具体 commit、文件和行号的审计报告

Git 审计任务默认不适合：

- 直接修改目标仓库
- 自动修复问题
- 只做格式或风格检查

## 创建时建议澄清的信息

- 目标仓库是本地路径还是远程 URL
- 审计范围：单个 commit、commit 区间、分支 HEAD、时间范围，还是定时增量审计
- 每次执行审计几个 commit；如果是定时任务，是否每次只审计一个 commit
- 首次执行从哪里开始：最新 commit、指定 hash、指定时间点或全部历史
- 如何处理无新 commit 的情况
- 是否允许 `git clone`、`git fetch`、`git pull`
- 是否只报告高风险，还是报告中风险和建议
- 报告需要包含哪些元信息，例如 hash、作者、时间、message、改动统计

## 跨次状态建议

如果这是定时增量审计，通常需要记录已审计进度，例如：

- `lastAuditedHash`
- 已审计 commit 集合
- 上次同步时间

这些属于任务业务状态。

规则：

- 不要把审计进度写回任务定义材料
- 定时任务执行期会由系统注入 schedule shared-state 目录
- task.md 中可以写“使用系统注入的 schedule shared-state 目录保存审计进度”
- task.md 中不要写死某个绝对路径
- 不要自造其他跨次状态文件路径
- 一次性任务通常不需要 shared-state，可直接按 `task.md` 指定范围执行

## task.md 推荐内容

可以自由组织，但建议包含：

- 仓库位置：本地路径或远程 URL
- 审计模式：单 commit、范围审计、定时增量审计
- commit 选择规则：如何确定本次要审计的 commit
- 状态规则：如果需要跨次状态，保存哪些字段，何时更新
- 审计维度：安全、逻辑、架构、性能、可观测性、提交质量等
- 风险等级：哪些必须报告，哪些作为建议
- 输出要求：报告结构、是否需要文件行号、修复建议、总结
- 成功标准：无新 commit、发现风险、状态更新等场景如何判断完成

## 审计维度建议

### 安全与合规

- 敏感信息硬编码：密码、Token、私钥、AccessKey、连接字符串
- 注入风险：SQL、命令、模板、XSS 等
- 权限与鉴权：新增接口是否缺少鉴权或权限控制降级
- 敏感数据暴露：日志、响应、错误信息中泄露敏感数据
- 依赖安全：新增依赖是否可信、版本是否锁定
- 非预期大文件、二进制、证书或数据文件提交

### 代码逻辑

- 空值、nil、越界、零值、负数、unicode 等边界
- 错误处理缺失或异常被吞掉
- 并发竞态、资源泄漏、锁/连接/文件未释放
- 状态机转换、条件判断、循环终止条件错误

### 架构与实现

- 职责边界混乱
- 新增不合理耦合或全局状态
- 抽象层泄漏
- 破坏公开接口兼容性
- 环境相关配置硬编码
- 过度复杂或重复造轮子

### 性能与可观测性

- N+1 查询或循环内网络/数据库请求
- 无限制查询或全量加载大对象
- 缓存策略退化
- 关键路径缺少日志、指标或错误上下文

### 提交质量

- commit message 是否清晰
- 是否混入不相关改动
- 是否容易 revert
- 是否缺少必要测试

## 报告结构建议

可参考以下结构，不强制：

```markdown
# Git Commit 审计报告

## Commit 信息

## 审计结论

## 风险汇总

## 高风险问题

## 中风险问题

## 建议

## 各维度结论

## 状态更新说明
```

## 不要继承的旧约定

- 不使用旧的独立状态目录
- 不要求固定保存到 `reports/` 目录
- 不要求在 `task.md` 中重复系统结果协议
- 不把管理器内部文件当作业务状态文件
