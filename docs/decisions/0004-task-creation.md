# 0004 Task Creation

日期：2026-04-22

## 结论

“创建任务”是系统内建能力，不属于普通任务执行流程。

- `drift task add` 先创建任务目录，再进入任务创建过程
- 创建任务支持三种方式：`claude`、`codex`、`manual`
- 创建完成后，由用户显式选择是否加入队列
- 不引入额外的 `draft` 状态；未入队任务只是尚未加入队列的任务目录

## 已确认规则

### 1. 创建任务的整体流程

`drift task add` 的流程如下：

```text
选择 TaskType
-> 选择创建方式
-> 创建任务目录
-> 进入创建过程（claude / codex / manual）
-> 检查 spec/task.md 是否有效
-> 选择执行 runner
-> 用户按需补充其他附加文件
-> 用户确认加入队列
-> 正式入队
```

### 2. 创建方式

支持三种创建方式：

- `claude`
- `codex`
- `manual`

规则：

- 只有系统中检测到 `claude` 命令时，才提供 `claude` 选项
- 只有系统中检测到 `codex` 命令时，才提供 `codex` 选项
- `manual` 始终可用

### 2.1 执行 runner 的选择

- 创建任务时，执行 runner 由用户显式选择
- 执行 runner 与创建方式是两个独立概念
- 创建方式决定“谁来辅助整理 `spec/task.md`”
- 执行 runner 决定后续任务真正入队执行时使用谁

约定：

- 执行 runner 的选择发生在 `spec/task.md` 创建并通过有效性检查之后
- 执行 runner 的候选项来自系统当前实际可用的 runner
- 例如只有系统检测到 `claude` / `codex` 命令时，才展示对应选项
- 用户选择的执行 runner 最终写入该任务的 `task.json.runner`

### 3. 创建目录与任务原件

在进入创建过程前，系统先创建任务目录。

最少需要准备：

```text
workspace/tasks/<taskId>/
  spec/
    task.md
```

其中：

- `taskId` 在创建任务目录时就生成
- 后续是否入队，都沿用同一个 `taskId`
- 不额外引入 `draft` 状态

### 4. 使用 agent 辅助创建

若选择 `claude` 或 `codex`：

- 系统先创建任务目录
- 然后进入对应 agent 的交互会话
- 会话的目标是帮助用户完成 `spec/task.md` 与任务材料整理
- 系统会为创建会话注入初始引导 prompt，明确当前目标是“创建任务”而不是“执行任务”
- 对于交互式 agent，会话启动时还应有一条简短可见的开场消息，引导 agent 先向用户询问需求，而不是静默等待
- 创建会话的工作目录是 `workspace/tasks/<taskId>/spec/`
- 会话结束后，程序返回主流程并检查 `spec/task.md`

约定：

- 任务创建会话不属于普通任务执行
- 不要求创建会话写 `agent-result.json`
- 不要求创建会话遵循执行期 `AgentResult` 协议
- 创建助手只围绕当前工作目录工作
- 注入给创建助手的 prompt 不应暴露 `spec/` 这个内部目录名，只称其为“当前目录”或“任务材料目录”
- 创建助手不应读取、搜索或引用当前工作目录之外的项目文件
- 创建助手不应向上级目录查找历史模板、旧任务、旧调度配置或示例文件
- 创建助手不应替任务管理器设计目录结构、状态文件或系统实现方案
- 如果用户讨论跨次执行记忆、进度游标、避免重复处理等问题，创建助手只应将其整理为任务需求
- 对于定时任务，创建助手可以说明执行期会由系统提供 schedule shared-state 目录，但不应写死绝对路径
- 创建助手不应建议将业务状态写回任务材料目录或其他系统状态路径
- 若该类型存在 guide，系统可以额外向创建助手提供该 guide 的绝对路径，且只作为只读补充材料
- 若创建会话中断，已创建的任务目录允许自然保留
- 第一版不要求 CLI 自动清理这类未形成有效 `task.md` 的孤儿目录

### 5. 手工创建

若选择 `manual`：

- 系统先创建任务目录
- 同时创建空的 `spec/task.md`
- 提示用户手工填写 `spec/task.md`
- 提示用户将其他附加文件放入 `spec/` 目录

### 6. 任务有效性校验

任务可加入队列的最小条件：

- `spec/task.md` 存在
- `spec/task.md` 内容非空
- 更严格地说，去掉空白字符后仍非空

也就是说：

- 只有空格、空行、换行的 `task.md` 视为无效
- 无效任务不可加入队列

### 7. 附加文件

- 附加文件由用户手工放入 `spec/` 目录即可
- 系统不要求理解附加文件的类型或语义
- 后续任务初始化时，`spec/` 会原样复制到 `workdir/`

### 8. TaskType 在创建阶段的作用

创建任务时，`TaskType` 的作用包括：

- 用于列出可选任务类型
- 提供展示名称与描述
- 提供默认执行配置
- 若该类型存在 type guide，可在创建阶段作为辅助参考材料

类型 guide 的作用：

- 提供该类任务通常需要澄清哪些信息的建议
- 不形成硬编码问卷
- 不覆盖具体任务的 `spec/task.md`
- 若存在 guide，则创建助手按系统提供的路径只读参考，不复制进任务目录

### 9. task.md 的组织方式

- `spec/task.md` 不要求固定格式
- 由创建助手或用户自由整理
- 系统可以有推荐结构，但不强制

### 10. 入队动作

- 创建任务与加入队列是两个动作
- 只有当用户明确选择“加入队列”时，任务才正式入队
- 若用户未选择加入队列，目录可以保留，不视为系统状态错误

## 原因

这样设计的目标是：

- 把“创建任务”和“执行任务”明确分开
- 保留用户对是否入队的控制权
- 避免把任务创建过程硬编码成固定表单
- 让 agent-assisted create 与 manual create 具备同等地位
- 保持任务正文对系统仍然是黑盒
